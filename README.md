## 🎯 Мета роботи
Ознайомитися з основами моделювання формальних специфікацій за допомогою інструменту Alloy Analyzer. Навчитися аналізувати моделі, знаходити логічні помилки за допомогою контрприкладів та розробляти власні специфікації предметних областей.

## 🧩 Виконані завдання та хід роботи

### Завдання 1: Рефакторинг моделі Person-Car
Початкова специфікація використовувала примітивний тип `String` для моделі машини (`c.model != ""`). Через відсутність обмежень SAT-розв'язувач Alloy згенерував контрприклад, де поле містило пробіли або пусті значення.
* **Рішення:** Проведено рефакторинг — впроваджено сувору об'єктну типізацію через сигнатури `Name` та `CarModel`. Повторна перевірка команди `check validCarAssignment` підтвердила повну ліквідацію дефекту коду ("No counterexample found").

### Завдання 2: Верифікація та розширення моделі бібліотеки
Розроблено базову модель, яка обмежує видачу книжок користувачам (макс. 3 одиниці одному користувачу, книга видається lone-користувачу).
* **Розширення:** Додано абстрактні статуси книги (`Available`, `Borrowed`), роль адміністратора (`Admin extends User`), а також числовий тип `Int` для суворого контролю терміну позичання книги (лічильник від 0 до 14 днів). Модель перевірена у розширеній області обчислень (`for 5 but 5 int`). Усі предикати зміни статусів адміном та ліміти часу успішно пройшли формальну верифікацію.

### Завдання 3: Модель соціальної мережі (Варіант 1)
Створено та верифіковано з нуля декларативну специфікацію соціальної мережі із відношенням `friends: set User` та фактами:
1. `MutualFriendship` — дружба є симетричною (якщо А друг Б, то і Б друг А).
2. `NoSelfFriends` — іррефлексивність (користувач не може дружити сам із собою).

За допомогою команд `run` та `check` доведено, що модель гнучко підтримує:
* Наявність циклічних "кіл друзів" (A-B-C-A).
* Існування ізольованих користувачів без друзів (спростовано `assert NoIsolatedUsers`).
* Топологічну декомпозицію мережі на декілька розрізнених ізольованих груп (спростовано твердження про повну зв'язність графа `AllUsersConnected`).

## ❓ Відповіді на контрольні питання
1. **Що таке Alloy?** Інструмент формального моделювання та верифікації специфікацій, що базується на декларативній логіці першого порядку і реляційному численні, призначений для пошуку помилок проєктування на ранніх етапах.
2. **Які основні компоненти мови Alloy?** Сигнатури (`sig`) — типи та відношення об'єктів; Факти (`fact`) — глобальні обов'язкові інваріанти системи; Предикати (`pred`) — параметризовані операції/поведінка; Твердження (`assert`) — властивості для перевірки командою `check`.
3. **Що таке "обмеження області исследования" (scope)?** Параметр (`for X`), який обмежує максимальну кількість екземплярів сигнатур у пошуку. Він робить простір станів скінченним для SAT-розв'язувача, прискорюючи аналіз (Bounded Model Checking).
4. **Що таке контрприклад?** Це конкретний згенерований стан або граф системи, який одночасно задовольняє всі глобальні факти (`fact`), але порушує висунуте розробником твердження (`assert`), що свідчить про логічну діру у специфікації.
